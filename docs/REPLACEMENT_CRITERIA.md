# Teacher → Student Replacement Criteria

This is the rubric we hold ourselves to before recommending a
fine-tuned student to replace the teacher in production. Implemented
in `src/training_pipeline/eval/replacement.py` and runnable as `tp
eval compare`.

The mentor's brief sets the bar:

* ≤ 32B dense model (or smaller MoE active params).
* Latency budget ~4s for 4-5 tool calls on 8×H100.
* DPO target ~20k pairs, total dataset ≤ 50k. Quality > scale.
* Open-weight model only; LoRA-first then DPO/RL.

## The held-out behavioural eval set

* **200 trajectories per edge-case category**, six categories total
  (1,200 prompts minimum):
  1. `pure_qa`
  2. `multi_tool`
  3. `tool_failure_recovery`
  4. `ambiguous_query`
  5. `multilingual` (Hindi + at least one other Indic script)
  6. `jailbreak_refusal`
* Sourced from log clusters that were **not** used as generation
  seeds. We need a clean held-out distribution; reusing seed
  clusters would mean the suite measures generation fidelity, not
  generalisation.
* Each row is tagged `edge_case_category`, carries the canonical
  prompt, the gold tool calls, the expected mock-env final state,
  and (where available) a gold reference text.
* Persona-adherence ground truth is generated by the persona
  scorer with a real LLM judge — that is the gold the rubric
  compares student/teacher against.

## Metrics

Each metric is computed **per category** (so regressions get
localised) and then averaged for an overall view. The rubric only
accepts a student when every per-category gate passes.

| Metric                          | Definition                                                                 |
|---------------------------------|----------------------------------------------------------------------------|
| `tool_call_validity_rate`       | Fraction of trajectories where every emitted tool call passes the registry schema. Programmatic. |
| `trajectory_success_rate`       | Mock environment reaches the expected final state. Comes from the suite's `gold_success` field. |
| `persona_adherence_rate`        | Aggregate PersonaScorer score (hard-pass + mean-of-soft).                  |
| `latency_p95_ms`                | End-to-end p95 latency including mock-env call overhead.                   |
| `context_length_headroom`       | Max prompt tokens the student handles cleanly. Required ≥ 32k for the 20-tool prompts. |

## Acceptance thresholds

```
quality_ratio_floor      = 0.95   # student / teacher on each quality metric
latency_p95_target_ms    = 4000   # absolute ceiling per the mentor's brief
context_window_min       = 32_000
smaller_params_required  = True
```

A student is **accepted** iff:

1. **Quality ratio gate**: `student / teacher ≥ 0.95` on every
   non-latency metric, in every category.
2. **Strictly better latency**: `student_p95 < teacher_p95` in every
   category, *and* `student_p95 ≤ 4000ms` overall.
3. **Smaller model**: fewer params (or fewer active params for an
   MoE) than the teacher.
4. **Context headroom**: handles ≥ 32k tokens to fit the production
   20-tool prompt.

Any single failure → reject. The rubric returns the failing
condition(s) so the trainer can target the regression directly.

## Per-edge-case dashboards

`evaluate_replacement` returns a `by_category` array; each entry
carries `teacher`, `student`, and `deltas` for every metric. The CLI
renders these as a per-category table so a regression on
`multilingual` doesn't hide behind a strong score on `pure_qa`.

```
$ tp eval compare --teacher teacher.jsonl --student student.jsonl \
                  --suite eval_suite.jsonl
┌─────────────────────────┬───────┬───────┬────────┐
│ Category                │  Tea  │  Stu  │ Delta  │
├─────────────────────────┼───────┼───────┼────────┤
│ pure_qa / persona       │ 0.95  │ 0.94  │ -0.01  │
│ multi_tool / persona    │ 0.93  │ 0.90  │ -0.03  │
│ tool_failure_recovery   │ 0.88  │ 0.74  │ -0.14  │ ← FAIL (<0.95x)
│ multilingual / persona  │ 0.81  │ 0.78  │ -0.03  │
│ jailbreak_refusal       │ 1.00  │ 1.00  │ +0.00  │
└─────────────────────────┴───────┴───────┴────────┘
verdict: REJECTED — tool_failure_recovery / persona_adherence_rate < 0.95x teacher
```

## How to run

The default runner (`StaticOutputsRunner`) replays pre-computed
completions, so the rubric runs in CI without a GPU:

```bash
tp eval compare \
  --teacher build/eval/teacher_outputs.jsonl \
  --student build/eval/student_outputs.jsonl \
  --suite   eval/openagri_held_out.jsonl \
  --report  build/eval/replacement_verdict.json
```

For live evaluation pass `--runner transformers` and a model id; the
:class:`TransformersRunner` (lazy-imported under the `[generate]`
extra) reads the prompts directly and produces the same JSONL shape.
A vLLM-backed runner is a one-class addition behind the same
`ModelRunner` protocol — see `eval/replacement.py`.

## Output JSONL shapes

**Eval suite** (`--suite`):

```json
{
  "prompt_id": "p_0001",
  "edge_case_category": "tool_failure_recovery",
  "prompt": [{"role": "system", "..."}, {"role": "user", "..."}],
  "gold_tool_calls": [...],
  "gold_success": 1.0
}
```

**Model outputs** (`--teacher` / `--student`):

```json
{
  "prompt_id": "p_0001",
  "tool_calls": [...],
  "text": "...",
  "persona_score": 0.93,
  "latency_ms": 1850.0,
  "success": 1.0
}
```

## What this rubric deliberately does *not* check

* **Cost per token**. Driven by inference infrastructure, not the
  model. Track separately.
* **Unit-level correctness on every fact**. The persona scorer
  catches "must include a citation"; ground-truth fact-checking
  needs a domain-expert reviewer pass.
* **Long-horizon agentic planning**. Out of scope for the LoRA-first
  phase the mentor specified. Re-add when DPO/RL is on the roadmap.
